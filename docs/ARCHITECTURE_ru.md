## Многоуровневая архитектура

### 1. Базовый компонентный слой (base/)

**Обязанность**: Предоставить самые фундаментальные компоненты, от которых зависят все остальные уровни.

**Компоненты**:
- `exceptions.py`: Пользовательские классы исключений
- `constants.py`: Определения констант
- `config_manager.py`: Управление конфигурацией
- `config_validator.py`: Валидация конфигурации

**Зависимости**: Нет (самый нижний уровень)

**Зависят от него**: Все остальные уровни

---

### 2. Слой моделей данных (models/)

**Обязанность**: Определение структур данных и моделей.

**Компоненты**:
- `conversation_models.py`: Модели данных Message, Session, MemoryEvent и другие.

**Зависимости**: base/

**Зависят от него**: managers/, processors/, storage/

---

### 3. Слой обработчиков (processors/)

**Обязанность**: Обработка и преобразование данных.

**Компоненты**:
- `memory_processor.py`: Использование LLM для обработки истории разговоров.
- `text_processor.py`: Токенизация и обработка текста.
- `chatroom_parser.py`: Парсинг контекста чат-комнаты.
- `message_utils.py`: Утилиты обработки сообщений.

**Зависимости**: base/, models/

**Зависят от него**: managers/, event_handler.py

---

### 4. Слой системы поиска (retrieval/)

**Обязанность**: Реализация функциональности поиска памяти.

**Компоненты**:
- `dual_route_retriever.py`: Координация финального ранжирования документного пути и графового пути.
- `hybrid_retriever.py`: Гибридный поиск по документному пути (BM25 + вектор).
- `bm25_retriever.py`: Поиск по ключевым словам по документному пути.
- `vector_retriever.py`: Векторный поиск по документному пути.
- `graph_keyword_retriever.py`: Графовый поиск по ключевым словам.
- `graph_vector_retriever.py`: Графовый векторный поиск.
- `graph_retriever.py`: Графовый фьюжн-ранжирование.
- `rrf_fusion.py`: Алгоритм слияния RRF.

**Зависимости**: base/, processors/text_processor

**Зависят от него**: managers/memory_engine

---

### 5. Слой менеджеров (managers/)

**Обязанность**: Управление основной бизнес-логикой.

**Компоненты**:
- `memory_engine.py`: Движок памяти (ядро).
- `conversation_manager.py`: Управление разговорами.

**Зависимости**: base/, models/, processors/, retrieval/

**Зависят от него**: plugin_initializer, event_handler, command_handler

---

### 6. Слой валидаторов (validators/)

**Обязанность**: Валидация данных и индексов.

**Компоненты**:
- `index_validator.py`: Валидация целостности индексов.

**Зависимости**: base/, managers/

**Зависят от него**: plugin_initializer, command_handler

---

### 7. Слой утилит (utils/)

**Обязанность**: Предоставление общих вспомогательных функций и форматировщиков.

**Компоненты**:
- `stopwords_manager.py`: Управление стоп-словами.
- `FakeToolCallFormatter` (`__init__.py`): Форматировщик поддельных вызовов инструментов.
  - Оборачивает содержимое воспоминаний в пары сообщений `tool_calls` + `tool`.
  - Использует фиксированный префикс `fake_recall_` в качестве call_id, что позволяет `EventHandler` автоматически очищать.
  - Поддерживает усечение по бюджету токенов для предотвращения превышения лимита контекста.

**Зависимости**: base/

**Зависят от него**: processors/, retrieval/, event_handler.py

---

### 8. Слой инструментов Agent (tools/)

**Обязанность**: Предоставление бизнес-инструментов, которые можно активно вызывать в режиме tool loop / agent AstrBot.

**Компоненты**:
- `memory_search_tool.py`: Инструмент проактивного восстановления долгосрочной памяти, повторно использующий существующий движок поиска памяти и конфигурацию фильтрации.

**Зависимости**: base/, managers/, utils/

**Зависят от него**: main.py (регистрация в контексте AstrBot)

---

### 9. Слой хранения (storage/)

**Обязанность**: Персистентность данных и резервное копирование.

**Компоненты**:
- `conversation_store.py`: Хранилище данных разговоров.
- `db_migration.py`: Миграция базы данных.
- `backup_manager.py`: Менеджер регулярного автоматического резервного копирования.
  - Планирование бэкапов на основе интервалов (cron / asyncio sleep).
  - Настраиваемая политика хранения (по дням или количеству).
  - Автоматический повтор и уведомление при сбое.

**Зависимости**: base/, models/

**Зависят от него**: managers/conversation_manager, plugin_initializer

---

### 10. Компоненты верхнего уровня

**Компоненты**:
- `plugin_initializer.py`: Инициализация плагина.
- `event_handler.py`: Обработка событий.
- `command_handler.py`: Обработка команд.
- `main.py`: Регистрация плагина.
- `tools/`: Источник реализации точки входа регистрации инструментов Agent.

**Зависимости**: Все компоненты нижних уровней.

**Зависят от него**: Нет (высший уровень)

## Принципы проектирования

### 1. Принцип единственной ответственности (SRP)

Каждый модуль отвечает только за одну чёткую функциональную область:
- base/: Инфраструктура
- models/: Структуры данных
- processors/: Обработка данных
- managers/: Бизнес-логика
- validators/: Логика валидации

### 2. Принцип инверсии зависимостей (DIP)

Высокоуровневые модули не зависят от низкоуровневых; оба зависят от абстракций:
- Использование интерфейсов и абстрактных классов.
- Передача зависимостей через внедрение зависимостей.

### 3. Принцип открытости/закрытости (OCP)

Открыты для расширения, закрыты для модификации:
- Новые функции реализуются добавлением новых модулей.
- Существующий стабильный код не модифицируется.

### 4. Принцип разделения интерфейса (ISP)

Клиенты не должны зависеть от интерфейсов, которые они не используют:
- Каждый модуль экспортирует только необходимые интерфейсы.
- Используйте `__all__` для явного определения экспорта.

### 5. Закон Деметры (LoD)

Модули должны взаимодействовать только со своими непосредственными зависимостями:
- Избегайте межуровневых вызовов.
- Передавайте запросы через промежуточные уровни.

## Матрица ответственности модулей

| Уровень | Ответственность | Пример |
|------|------|------|
| base/ | Инфраструктура | Исключения, конфигурация, константы |
| models/ | Определения данных | Message, Session |
| processors/ | Обработка данных | Обработка текста, вызовы LLM |
| retrieval/ | Реализация поиска | Документный путь, графовый путь, слияние RRF |
| managers/ | Бизнес-логика | Управление памятью, управление разговорами |
| validators/ | Логика валидации | Валидация индексов |

| storage/ | Персистентность данных | Операции SQLite |

### Архитектура фронтенда WebUI

**Файлы**:
- `index.html`: Каркас одностраничного приложения с переключателем языков и тёмной темы.
- `app.js`: Основная логика дашборда (таблица воспоминаний, поиск, пагинация, CRUD).
- `graph-ui.js`: 3D-рендерер графа знаний.
  - Форс-граф на основе `ForceGraph3D` (Three.js).
  - Поддержка перетаскивания узлов, масштабирования, вращения, всплывающих подсказок.
  - Раскраска по типам сущностей: человек, место, событие, концепция, другое.
  - Общается с `app.js` через шину событий, разделяя критерии фильтрации.
- `i18n.js`: Движок интернационализации.
  - Поддержка `zh` / `en` / `ru` (русский как fallback).
  - Перевод DOM на основе атрибутов `data-i18n`.
  - Постоянное хранение в `localStorage` + автоматическое определение `navigator.language`.
- `styles.css`: Дизайн-система в стиле glassmorphism.

## Руководство по расширению

### Добавление нового обработчика

1. Создать новый файл в `core/processors/`.
2. Реализовать класс обработчика.
3. Экспортировать его в `core/processors/__init__.py`.
4. Импортировать и использовать там, где нужно.

### Добавление нового поискового движка

1. Создать новый файл в `core/retrieval/`.
2. Реализовать интерфейс поискового движка.
3. Интегрировать его в `HybridRetriever`.
4. Обновить конфигурацию и документацию.

### Добавление нового менеджера

1. Создать новый файл в `core/managers/`.
2. Реализовать класс менеджера.
3. Экспортировать его в `core/managers/__init__.py`.
4. Интегрировать его в инициализатор.

## Лучшие практики

### 1. Порядок импортов

```python
# Стандартная библиотека
import os
import time

# Сторонние библиотеки
import aiosqlite
from astrbot.api import logger

# Локальные импорты — в порядке уровней
from ..base import ConfigManager, LivingMemoryException
```

### 2. Избежание циклических зависимостей

- Используйте аннотации типов в виде строк.
- Используйте ленивые импорты (импорт внутри функций).
- Перепроектируйте границы модулей.

### 3. Проектирование интерфейсов

- Используйте аннотации типов.
- Предоставляйте докстринги.
- Определяйте параметры и возвращаемые значения.

### 4. Обработка ошибок

- Используйте пользовательские исключения.
- Перехватывайте на подходящем уровне.
- Предоставляйте чёткие сообщения об ошибках.

## История версий

### v2.2.10

- **Инъекция через поддельный вызов инструмента**: Добавлена стратегия инъекции поддельного вызова инструмента, совместимая с режимом Agent / Tool Loop.

- **Регулярный авто-бэкап**: Добавлена подсистема регулярного автоматического резервного копирования (`backup_manager.py`).
